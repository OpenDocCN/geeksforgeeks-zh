# 了解登录屏幕后面的网页认证

> 原文: [https://www.geeksforgeeks.org/understanding-web-authentication-behind-the-login-screen/](https://www.geeksforgeeks.org/understanding-web-authentication-behind-the-login-screen/)

[**认证**](https://www.geeksforgeeks.org/authentication-in-computer-network/) 是验证用户身份或信息的过程。用户身份验证是在用户登录到计算机系统时验证用户身份的过程。

本文简要介绍了现代 web 应用程序中的身份验证。

两种主要的身份验证类型是:

*   无状态认证
*   状态认证

要了解两者之间的显著差异，请查看无状态和有状态协议之间的[差异文章](https://www.geeksforgeeks.org/difference-between-stateless-and-stateful-protocol/)。这两种类型的认证都可以通过理解它们的流程来理解。

## 状态认证

使用会话标识和 Cookies。

### 认证流程

1.  用户提交登录凭据，即用户名和密码。
2.  服务器根据数据库验证这些凭据。
3.  然后服务器生成一个临时的用户会话。
4.  服务器颁发一个带有会话 ID 的 Cookie。
5.  用户/客户端可以在每个请求中发送 Cookie。
6.  服务器根据会话存储进行身份验证并授予访问权限。
7.  当用户注销时，服务器销毁会话并清除 Cookie。

**注意:** 会话 ID 只不过是一个随机字符串，服务器可以根据会话的存储来识别它，以验证用户的请求。因此，没有第三方可以从该会话中提取任何数据/信息。

### 有状态身份验证的特性

每个用户会话都存储在服务器端(有状态)。

*   内存，即文件系统 (较少见)。
*   数据库缓存 (如 `Redis` 或 `Memcached`)。
*   (常用的包括 `Postgres` 和 `MongoDb`)。

**每个用户由会话标识来标识。**

1.  **不透明引用:**
    *   第三方无法提取任何数据。
    *   只有颁发方（服务器）可以映射回数据。
2.  **存储在 Cookie 中:**
    *   签名保密。此外，Cookie 通常受标志保护，因此客户端无法篡改 Cookie。服务器端对 Cookie 进行加密和签名的过程也是为了避免/检测客户端 Cookie 的任何篡改，因此服务器端可以确保 Cookie 未被篡改。

## 无状态认证

使用令牌，`JWT`，`OAuth` & 其他。

### 使用令牌的身份验证流程

*   用户提交登录凭据，即用户名和密码。
*   服务器根据数据库验证凭据。
*   然后服务器生成一个临时令牌并将用户数据嵌入其中。
*   响应服务器令牌（在正文或头部中）。
*   用户令牌存储在客户端存储 [`localStorage` 或 `SessionStorage`] 中。
*   在每个请求中发送用户令牌。
*   服务器验证令牌并授予访问权限。
*   当用户注销时，令牌从客户端存储中清除。

### 无状态认证系统的特性

*   令牌不存储在服务器端，只存储在客户端。
*   （无状态）签名保密以防止篡改。
*   通过验证，可以被服务器信任。
*   令牌可以是不透明的或独立的。
*   令牌可以在其载荷中携带所有所需的用户数据。
*   这可以减少数据库查找，但会将数据暴露给 `XSS` 攻击。
*   令牌通常在 `Authorization` 头部下发送。
*   当令牌即将过期时，可以刷新它，因为客户端同时获得访问令牌和刷新令牌。
*   刷新令牌可用于刷新访问令牌。
*   用于 `SPA` 网络应用、网络 `API` 甚至移动应用。

```
// 示例：一个简单的令牌验证逻辑（伪代码）
if (token.isExpired()) {
    refreshToken();
}
```
