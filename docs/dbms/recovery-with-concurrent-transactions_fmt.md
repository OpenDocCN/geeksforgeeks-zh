# 并发事务恢复

> 原文: [https://www.geeksforgeeks.org/recovery-with-concurrent-transactions/](https://www.geeksforgeeks.org/recovery-with-concurrent-transactions/)

[并发控制](https://www.geeksforgeeks.org/concurrency-control-in-dbms/)是指可以同时执行多个事务，然后出现交错的日志。但是事务结果可能会有变化，所以请保持这些事务的执行顺序。

在恢复过程中，恢复系统很难回溯所有日志，然后开始恢复。

并发事务的恢复可以通过以下四种方式完成。

1.  与并发控制的交互
2.  事务回滚
3.  检查点
4.  重启恢复

## 与并发控制的交互

在该方案中，恢复方案在很大程度上取决于所使用的并发控制方案。因此，要回滚失败的事务，我们必须撤销事务执行的更新。

## 事务回滚

*   在此场景中，我们使用日志来回滚失败的事务。
*   系统在失败的事务之后反向扫描日志，对于在日志中找到的每条日志记录，系统将恢复该数据项。

## 检查点

*   检查点是保存应用程序状态快照的过程，以便在发生故障时可以从该点重新启动。
*   检查点是将记录从缓冲区写入数据库的时间点。
*   检查点缩短了恢复过程。
*   当到达检查点时，事务将被更新到数据库，在此之前，整个日志文件将从文件中删除。然后用新的事务步骤更新日志文件，直到下一个检查点，依此类推。
*   检查点用于声明数据库管理系统处于一致状态且所有事务都已提交的点。

为了缓解这种情况，大多数数据库管理系统都使用了“[检查点](https://www.geeksforgeeks.org/log-based-recovery-in-dbms/)”概念。

*   在此方案中，我们使用检查点来减少系统从崩溃中恢复时必须扫描的日志记录数量。
*   在并发事务处理系统中，我们要求检查点日志记录的形式为，其中`'l'`是在检查点处于活动状态的事务列表。
*   模糊检查点是允许事务执行更新的检查点，即使在缓冲区块被写出时也是如此。

## 重启恢复

*   当系统从崩溃中恢复时，它将构建两个列表。
*   撤销列表由需要撤销的事务组成，重做列表由需要重做的事务组成。
*   系统按如下方式构建这两个列表：最初，它们都是空的。系统反向扫描日志并检查每条记录，直到找到第一条记录。
