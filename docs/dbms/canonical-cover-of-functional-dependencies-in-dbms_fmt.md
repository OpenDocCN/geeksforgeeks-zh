# 数据库管理系统中函数依赖的规范覆盖

> 原文：[https://www.geeksforgeeks.org/canonical-cover-of-functional-dependencies-in-dbms/](https://www.geeksforgeeks.org/canonical-cover-of-functional-dependencies-in-dbms/)

每当用户更新数据库时，系统必须检查在这个过程中是否有任何功能依赖被违反。如果在新的数据库状态中违反了依赖关系，系统必须回滚。使用大量的[函数依赖](https://www.geeksforgeeks.org/functional-dependency-and-attribute-closure/)会导致不必要的额外计算时间。这就是规范封面发挥作用的地方。
一组函数依赖 `F` 的规范覆盖是一组简化的函数依赖，其闭包与原始集合 `F` 相同。

## 重要定义

### 无关属性
如果我们可以在不改变函数依赖集的闭包的情况下移除一个函数依赖的属性，那么这个属性就被称为无关属性。

### 规范覆盖
一组函数依赖关系 `F` 的规范覆盖 `F_{c}`，从而满足以下所有属性：

*   `F` 在逻辑上意味着 `F_{c}` 中的所有依赖关系。
*   `F_{c}` 逻辑上隐含 `F` 中的所有依赖关系。
*   `F_{c}` 中没有包含无关属性的函数依赖。
*   `F_{c}` 中函数依赖的每一个左侧都是唯一的。也就是说 `α_{1} → β_{1}` 和 `α_{2} → β_{2}` 之间没有两个依赖关系 `α_{1} → α_{2}`。

## 寻找规范覆盖

### 计算集合 `F` 的典范覆盖的算法

```
repeat
    1. Use the union rule to replace any dependencies in F with a single dependency.
    2. Find a functional dependency X -> Y in F with an extraneous attribute either in X or in Y.
    3. If an extraneous attribute is found, delete it from X -> Y.
until F does not change
```

### 示例 1

考虑以下功能依赖的集合 `F`：

```
F = {
    A -> BC
    B -> C
    A -> B
    AB -> C
}
```

寻找规范封面的步骤：

1.  有两个函数依赖的左侧属性集相同：
    `A -> BC`
    `A -> B`
    这两者结合起来就可以得到 `A -> BC`。
    现在，修改后的集合 `F` 变成：
    ```
    F = {
        A -> BC
        B -> C
        AB -> C
    }
    ```

2.  在 `AB -> C` 中存在一个无关属性，因为即使从集合 `F` 中移除 `AB -> C`，我们也能得到相同的闭包。这是因为 `B -> C` 已经是 `F` 的一部分。
    现在，修改后的集合 `F` 变成：
    ```
    F = {
        A -> BC
        B -> C
    }
    ```

3.  `C` 是 `A -> BC` 中的一个无关属性，并且 `A -> B` 可以通过 `A -> B` 和 `B -> C`（根据传递性）逻辑推导出来。
    ```
    F = {
        A -> B
        B -> C
    }
    ```

4.  经过这一步，`F` 不再变化。所以，
    因此需要的规范封面是，
    `F_{c}` = {
        `A -> B`
        `B -> C`
    }

### 示例 2

考虑另一组函数依赖：

```
F = {
    A -> BC
    CD -> E
    B -> D
    E -> A
}
```

1.  `F` 中每个函数依赖的左侧都是唯一的。
2.  任何函数依赖的左侧或右侧的属性都不是无关的（通过对每个函数依赖应用无关属性的定义来检查）。
3.  因此，规范覆盖 `F_{c}` 等于 `F`。

**注：** 一组 `F` 个函数依赖可以有多个典范覆盖 `F_{c}`。

## 如何检查一组函数依赖 `F` 是否规范地覆盖了另一组函数依赖 `G`？

考虑以下两组功能依赖关系：

```
F = {
    A -> B
    AB -> C
    D -> AC
    D -> E
}

G = {
    A -> B
    D -> AB
}
```

现在，我们需要找出这些函数依赖中的一个是否规范地覆盖了另一组函数依赖。这意味着，我们需要找出 `F` 是否规范地覆盖了 `G`，`G` 是否规范地覆盖了 `F`，或者两者都没有规范地覆盖另一个。

为了找到答案，我们遵循以下步骤：

*   **创建单元素右部。** 这意味着，函数依赖箭头右侧的属性都应该是单个属性。
    函数依赖 `D -> AC` 被分解为两个函数依赖：`D -> A` 和 `D -> C`。
    ```
    F = {
        A -> B
        AB -> C
        D -> A
        D -> C
        D -> E
    }
    ```

*   **移除所有无关属性。**
    考虑任何函数依赖 `XY -> Z`，如果 `X` 本身可以决定 `Z`，那么属性 `Y` 是无关的，可以去掉。正如我们所看到的，无关属性的出现只有在左侧有多个属性的函数依赖中才有可能。
    所以，考虑函数依赖 `AB -> C`。
    现在，我们必须找到 `A` 和 `B` 的闭包，以确定其中是否有任何一个是无关的。
    `[A]^{+}` = `AB`
    `[B]^{+}` = `B`
    我们可以看到，`B` 可以从 `A` 中确定，这意味着我们可以从函数依赖 `AB -> C` 中移除 `B`。
    ```
    F = {
        A -> B
        A -> C
        D -> A
        D -> C
        D -> E
    }
    ```

*   **移除所有冗余的函数依赖。**

## 检查函数依赖的冗余性

逐一检查所有的函数依赖（`f.d.`），看看通过移除一个函数依赖 `X -> Y`，我们是否仍然可以通过其他一些函数依赖从 `X` 中找出 `Y`。更正式的说法是在不使用该函数依赖的情况下找到 `X` 的闭包 `X+`。我们正在测试并检查 `Y` 是否是闭包的一部分。如果是的话，那么该函数依赖是多余的。

在这里，当检查函数依赖 `D -> C` 时，我们观察到即使在隐藏它之后，`D` 的闭包也包含 `C`，这是因为我们可以通过其他两个函数依赖 `D -> A` 和 `A -> C` 的组合从 `D` 中获得 `C`，所以，`D -> C` 是多余的。

```
f = {
    A -> B
    A -> C
    D -> A
    D -> E
}
```

现在，对 `g` 做同样的事情。

*   创建单例右部。这意味着，函数依赖箭头右侧的属性都应该是单例。
```
g = {
    A -> B
    A -> C
    D -> A
    D -> B
}
```

*   移除所有无关属性。
    由于所有函数依赖的右部（`RHS`）只包含 1 个属性，所以不可能有无关属性。

*   移除所有冗余的函数依赖。

通过遍历所有的函数依赖并在所有情况下检查左部（`LHS`）的闭包，我们观察到函数依赖 `D -> B` 是多余的，因为它可以通过两个其他函数依赖 `D -> A` 和 `A -> B` 的组合来获得。

```
g = {
    A -> B
    A -> C
    D -> A
}
```

现在，由于 `G` 的所有函数依赖已经包含在 `F` 中，我们得出结论 `F` 包含 `G`。
