# 软件工程|敏捷开发模型

> 原文：[https://www.geeksforgeeks.org/software-engineering-agile-development-models/](https://www.geeksforgeeks.org/software-engineering-agile-development-models/)

在早期，迭代瀑布模型在完成一个项目时非常流行。但是现在开发人员在使用它开发软件时面临着各种各样的问题。主要困难包括在项目开发期间处理客户的变更请求，以及合并这些变更所需的高成本和时间。为了克服瀑布模型的这些缺点，在20世纪90年代中期，敏捷软件开发模型被提出。

敏捷模型主要是为了帮助项目快速适应变更请求。因此，敏捷模型的主要目的是促进项目快速完成。要完成这项任务，需要敏捷性。敏捷性是通过将过程与项目相适应，删除对特定项目来说不重要的活动来实现的。此外，任何浪费时间和精力的事情都可以避免。

实际上敏捷模型指的是一组开发过程。这些过程有一些共同的基本特征，但它们之间确实有一些微妙的差异。下面给出了几个敏捷SDLC模型：

*   `Crystal`
*   `Function-driven development`
*   混乱
*   `Extreme programming (XP)`
*   `Lean development`
*   `Unified process`

在敏捷模型中，需求被分解成许多可以增量开发的小部分。敏捷模型采用迭代开发。每个增量部分都是通过迭代开发的。每一次迭代都是小的，易于管理的，并且只能在几周内完成。一次为客户规划、开发和部署一次迭代。没有制定长期计划。

敏捷模型是迭代和增量过程模型的结合。敏捷SDLC模型涉及的步骤有：

*   需求收集
*   需求分析
*   设计
*   编码
*   单元测试
*   验收测试

完成迭代的时间被称为`时间盒`。`时间盒`是指向客户交付迭代所需的最长时间。因此，迭代的结束日期不会改变。尽管开发团队可以决定在`时间盒`期间减少交付的功能，如果有必要的话，按时交付。敏捷模型的核心原则是在每个时间框之后向客户交付增量。

## 敏捷模型的原则：

*   为了在开发过程中与客户建立密切联系并清楚地了解各种需求，每个敏捷项目通常在团队中包含一名客户代表。在每次迭代结束时，利益相关者和客户代表会审查取得的进展并重新评估需求。
*   敏捷模型依赖于可工作软件的部署，而不是全面的文档。
*   每隔几周就向客户代表频繁交付增量版本的软件。
*   并鼓励有效整合客户需求变更请求。
*   拥有高效的团队成员并加强团队成员之间的沟通更为重要。人们认识到，开发团队成员之间增强的沟通可以通过面对面的交流实现，而不是通过正式文档的交换。
*   建议保持开发团队规模较小（5-9人），以帮助团队成员进行有意义的面对面沟通并拥有协作的工作环境。
*   敏捷开发过程通常采用`结对编程`。在`结对编程`中，两名程序员在一个工作站上一起工作。一个人编码，另一个人在输入代码时检查代码。两名程序员大约每小时交换一次角色。

## 优势：

*   `结对编程`产生编写良好、紧凑的程序，其错误比单独工作的程序员少。
*   整个项目的总开发时间减少。
*   每次迭代后，客户代表会获得软件产品更新的想法。因此，如果需要，他可以轻松更改任何需求。

## 缺点：

*   由于缺乏正式文档，导致混淆，在不同阶段做出的重要决策可能随时被不同的团队成员误解。
*   由于缺乏适当的文档，当项目完成且开发人员被分配到另一个项目时，开发项目的维护将成为一个问题。
