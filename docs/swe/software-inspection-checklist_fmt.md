# 软件检查清单

> 原文：[https://www.geeksforgeeks.org/software-inspection-checklist/](https://www.geeksforgeeks.org/software-inspection-checklist/)

[软件检查](https://www.geeksforgeeks.org/how-software-inspection-improves-software-quality/)流程通常需要检查表，以便在检查[软件产品](https://www.geeksforgeeks.org/software-engineering-software-product/)时向审查人员提供确定和识别缺陷的提示和建议。检查过程应该总是由一些常见编程错误的清单来驱动。

一个`检查清单`仅仅是对特定软件产品已被检查的保证。应通过与有经验的工作人员讨论来制定检查清单，并随着检查过程中获得更多经验而定期更新。指南通常包括各种工件的清单，如设计文件、需求等。

不同的`清单`也是针对各种[编程语言](https://www.geeksforgeeks.org/introduction-to-programming-languages/)准备的，即针对不同的源语言分别给出源代码清单。清单中有一些项目是一般性的，因此需要大量的人工判断。检查过程中可能进行的一些检查如下：

1.  **`数据失败`：**
    *   检查程序中的所有变量在使用前是否都已初始化？
    *   所有常量都有名称吗？
    *   缓冲区溢出的可能性有多大？
2.  **`控制失败`：**
    *   每个条件语句的条件是否正确？
    *   每个循环是否都必须结束？
    *   复合语句的括号是否正确？
3.  **`输入/输出（I/O）失败`：**
    *   所有输入变量都被使用了吗？
    *   所有输出变量在输出前都被赋值了吗？
    *   输入是否可能被意外破坏？
4.  **`接口失败`：**
    *   所有方法和函数的参数是否正确？
    *   参数的类型在实际和形式上是否匹配？
    *   参数的顺序是否正确？
    *   如果所有组件都访问共享内存，它们是否具有相同类型的共享内存结构？
5.  **`存储管理失败`：**
    *   如果链表结构被修改，所有链接是否都正确重新分配？
    *   如果使用了动态存储，空间是否被正确分配？
    *   当不再需要时，是否显式地释放了空间？
6.  **`异常管理失败`：**
    *   所有可能的错误情况是否都已被考虑？

## 示例

### 需求清单

*   需求与数据之间是否有明确的区分？
*   需求是否准确定义了需要呈现给用户的所有信息？
*   是否包含了系统和用户对所有错误情况的响应需求？
*   每个需求是否都表述清晰、简洁和明确？
*   每个需求都是可测试的吗？
*   是否存在任何模糊或隐含的需求？
*   是否存在相互冲突的需求？
*   软件需求规格说明书（SRS）中是否有待解决的问题？
*   性能需求（如响应时间、数据存储需求等）是否已声明？

### 错误处理与恢复清单

*   是否有足够的错误条件用于测试？
*   是否测试了那些发生概率高或错误结果对系统致命的错误情况？
*   所有返回码都被记录了吗？
*   所有返回信息都能被理解吗？
*   程序是否允许成功的错误恢复？这是否跨越模块或进程？跨越操作系统？跨越断点？硬件故障？
